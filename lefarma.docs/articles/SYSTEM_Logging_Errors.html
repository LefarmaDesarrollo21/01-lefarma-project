📊 Sistema de Observabilidad: WideEvents y Persistencia de Errores

Lefarma implementa una arquitectura de logging de alta fidelidad basada en el patrón Wide Event (un evento único y rico por cada petición HTTP). Este sistema no solo registra texto, sino que captura el estado completo de la aplicación, el contexto del usuario y el resultado del negocio en un formato estructurado (JSON).

🏗️ Anatomía de un WideEvent

Cada objeto WideEvent se compone de cinco dimensiones de datos:

DimensiónCampos PrincipalesPropósito
Identificación RequestId, TraceId, SpanId Correlación entre microservicios y rastreo de la petición única.
Contexto HTTP Method, Endpoint, IpAddress, UserAgent Detalles técnicos de la conexión y origen del cliente.
Contexto de Usuario UserId, UserSubscription Identifica quién realizó la acción (obtenido de los JWT claims).
Contexto de Negocio EntityType, EntityId, Action, AdditionalContext Explica qué recurso de negocio se afectó (ej: Orden #105, Autorizar).
Resultado StatusCode, DurationMs, Status (Success/Error) Métricas de performance y éxito técnico de la operación.

💾 Política de Persistencia en Base de Datos

Aunque el 100% de los WideEvents se emiten a los logs de archivos (Serilog JSON), solo una parte crítica se persiste en la tabla ErrorLogs de SQL Server:

  • Errores de Servidor (5xx): Cualquier respuesta con código 500 o superior es persistida automáticamente.
  • Excepciones No Controladas: Cualquier excepción que rompa el pipeline y sea capturada por el WideEventLoggingMiddleware.
  • Captura de Payloads: En caso de error en peticiones POST/PUT, el sistema intenta capturar los primeros 8KB del RequestBody para facilitar el debugging.
Performance: El guardado en base de datos se realiza mediante un hilo en segundo plano (Fire-and-Forget) para no penalizar el tiempo de respuesta al usuario.

🗃️ Esquema de la Tabla ErrorLogs

La entidad ErrorLog mapea el WideEvent a una estructura relacional optimizada para consultas:

🛠️ Guía para Desarrolladores: Enriquecimiento

El poder del WideEvent reside en su capacidad de ser enriquecido en cualquier parte del código usando el IWideEventAccessor.

// 1. Inyectar el accesor
private readonly IWideEventAccessor _log;

public async Task ProcesarPago(int facturaId) {
    // 2. Añadir contexto de negocio ANTES de que ocurra el error
    _log.Current?.Enrich("Factura", facturaId, "ProcesarPago")
                 .AddContext("ProveedorId", 123)
                 .AddContext("Monto", 5000.50);

    // Si ocurre una excepción después de aquí, el ErrorLog 
    // en la BD tendrá asociados estos datos de factura.
}

🔍 Análisis de Trazabilidad

Para buscar un error reportado por un usuario:

  1. Solicita el RequestId o ErrorGuid (si se muestra en el frontend).
  2. Consulta la base de datos: SELECT * FROM logging.ErrorLogs WHERE RequestId = '...'
  3. Revisa el StackTrace y los DatosAdicionales para recrear el escenario.
